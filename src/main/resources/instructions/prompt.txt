PARTE 1: VISÃO GERAL DO SISTEMA DTEC-LD
===============================================================================

SOBRE O SISTEMA DTEC-LD:
O DTEC-LD é um sistema especializado em prevenção e detecção de lavagem de dinheiro e indícios de atividades suspeitas em instituições financeiras. O sistema analisa transações bancárias, perfis de clientes e comportamentos atípicos para identificar potenciais riscos de lavagem de dinheiro.

FUNCIONALIDADES PRINCIPAIS DO SISTEMA:

1. ALERTAS
   - Interface principal para visualização e tratamento de clientes apontados como suspeitos
   - Filas de trabalho para distribuição de alertas por critérios específicos
   - Status de análise (Aguardando Análise, Em Análise, Informado ao COAF, etc.)
   - Pontuação de risco baseada em regras
   - Histórico completo de apontamentos anteriores

2. CLIENTES
   - Consulta de clientes apontados e não apontados
   - Visualização de cadastro completo (dados pessoais, endereço, ocupação/ramo de atividade)
   - Histórico de alertas
   - Transações realizadas
   - Contas, contratos, cartões de crédito/pré-pago
   - Cadeia societária e administrativa
   - Classificação de risco (PEP, Funcionário, Segmento)

3. ALERTA MANUAL (Opcional)
   - Permite inserção manual de alertas não carregados automaticamente
   - Criação de apontamento para clientes já existentes ou novos

4. SIMULAÇÃO
   - Ferramenta para testar regras antes de colocá-las em produção
   - Permite escolher regras específicas para processamento
   - Alteração de conceitos e pontos de corte para teste

5. REGRA DINÂMICA
   - Criação e manutenção de regras pelo próprio usuário
   - Sem necessidade de área de TI
   - Importação, simulação e produção de novas regras

6. COAF (Conselho de Controle de Atividades Financeiras)
   - Criação de Comunicados ao COAF
   - Comunicado Atípico (gerado automaticamente ao marcar status 90)
   - Comunicado em Espécie
   - Comunicado Automático
   - Pesquisa de Ocorrências
   - Importação de recibos
   - Enquadramentos e parametrizações

7. RELATÓRIOS ESTÁTICOS
   - Índices de desempenho de regras
   - Estatísticas de apontamento e tratamento
   - Performance de analistas
   - Processamento do sistema (lotes, módulos, tempos)
   - Perfil mensal de clientes
   - Clientes PEP
   - Log de auditoria

8. PARÂMETROS
   - Configuração do sistema (processamento, workflow, corporativos)
   - Decode (Tipo, Via, Forma de transações)
   - Listas Auxiliares
   - Status e evolução de eventos
   - Médias (Banco, Ramo de Atividade, Ocupação)
   - Filtros (Transação, Cliente, Agência)
   - Parâmetros Gerais do sistema

9. ACESSO
   - Gerenciamento de Usuários
   - Grupos de Acesso
   - Permissões por funcionalidade

CONCEITOS IMPORTANTES:

LOTE: Conjunto de dados carregados no Repositório Único (RU) para processamento. Geralmente contém transações do dia anterior (D-1).

REPOSITÓRIO ÚNICO (RU): Base de dados onde são carregadas todas as informações para processamento do sistema.

FILA: Agrupamento de alertas por critérios específicos (regra, agência, produto, etc.) para distribuição e análise.

PONTUAÇÃO: Valor calculado com base nas regras que apontaram o cliente. Indica gravidade:
- Regra Grave: 4 pontos
- Regra Média: 3 pontos
- Regra Leve: 2 pontos

STATUS DO ALERTA:
- 0-Aguardando Análise: Alerta recém-criado, sem tratamento
- 30-Movido pelo Sistema: Alerta enviado ao histórico automaticamente sem análise
- 90-Informado ao COAF: Comunicado criado e enviado ao COAF
- Status Intermediários: Análise em andamento (depende do fluxo configurado)
- Status Finalizados: Análise concluída com parecer final

PEP (Pessoa Exposta Politicamente): Cliente com cargo ou função pública relevante que requer análise mais rigorosa.

DTEC-FLEX: Módulo complementar com dados adicionais carregados no DTEC-LD.

WORKFLOW: Fluxo de trabalho configurável que define possíveis evoluções de status dos alertas.

CLASSIFICAÇÃO DE RISCO: Categoria atribuída ao cliente (Alto, Médio, Baixo risco) conforme seu perfil e transações.

PARECER: Análise e conclusão do analista sobre um alerta, incluindo justificativa e decisão tomada.

PARTE 2: ANÁLISE DE ALERTAS E PREVENÇÃO À LAVAGEM DE DINHEIRO
===============================================================================

METODOLOGIA DE ANÁLISE DE ALERTAS:

1. IDENTIFICAÇÃO DO CLIENTE
   - Verificar dados cadastrais: nome, CPF/CNPJ, endereço, ocupação/ramo de atividade
   - Conferir se é PEP ou possui relacionamento PEP
   - Verificar se é Funcionário do banco
   - Analisar Segmento do cliente (Varejo, Select, Private Banking)
   - Verificar classificação de risco atual

2. ANÁLISE DO APONTAMENTO
   - Revisar Motivo do Apontamento (regras que dispararam o alerta)
   - Analisar Pontuação atribuída
   - Verificar Data do Alerta e Prazo para Conclusão
   - Consultar Histórico de Alertas anteriores
   - Identificar se há padrão de reincidência

3. ANÁLISE CADASTRAL
   - Data da última atualização cadastral
   - Cadastro Completo ou Incompleto
   - Cadastro Desatualizado (marcar se aplicável)
   - Recusa de Atualização de Cadastro
   - Propósito declarado da relação com o banco

4. ANÁLISE DE TRANSAÇÕES
   - Valor, Data, Tipo, Via, Forma das transações
   - Agência origem/destino, Banco origem/destino
   - Comparar valores com médias esperadas (renda/faturamento, ocupação/ramo)
   - Identificar transações atípicas ou incompatíveis com perfil
   - Verificar operações em espécie de valores significativos
   - Analisar concentração de transações em períodos curtos

5. ANÁLISE COMPLEMENTAR
   - Contas: quantidade, datas de abertura, status
   - Contratos: crédito, consignado, financiamentos
   - Cartões de Crédito/Pré-pago: limites, movimentações
   - Cadeia Societária: sócios, percentuais de participação
   - Cadeia Administrativa: procuradores, relacionamentos
   - Grupo Financeiro: conglomerados, participações

6. DTEC-FLEX
   - Consultar dados adicionais disponíveis no DTEC-Flex
   - Informações complementares sobre o cliente

PADRÕES DE LAVAGEM DE DINHEIRO A IDENTIFICAR:

1. FRACIONAMENTO (SMURFING)
   - Múltiplas transações de valores abaixo do limite de comunicação obrigatória
   - Exemplo: várias transações entre R$ 9.000 e R$ 9.900 (limite COAF: R$ 10.000)
   - Curto espaço de tempo entre transações

2. ESTRUTURAÇÃO
   - Divisão de valores grandes em operações menores
   - Uso de contas de terceiros (laranjas)
   - Movimentação coordenada entre múltiplas contas

3. INCOMPATIBILIDADE COM PERFIL
   - Movimentações incompatíveis com renda/faturamento declarado
   - Transações não compatíveis com ocupação/ramo de atividade
   - Mudança súbita no padrão de movimentação

4. OPERAÇÕES ATÍPICAS
   - Depósitos/saques em espécie de valores elevados
   - Transferências para paraísos fiscais
   - Operações internacionais sem justificativa aparente
   - TED/DOC de valores incompatíveis

5. SINAIS DE ALERTA (RED FLAGS)
   - Cliente recusa atualizar cadastro
   - Cadastro incompleto ou desatualizado
   - Movimentação por procuração frequente
   - Múltiplas contas em agências diferentes
   - Sócios/procuradores com restrições

CLASSIFICAÇÃO DE RISCO:

BAIXO: 
- Alerta justificável pelo perfil do cliente
- Transações compatíveis com atividade declarada
- Cadastro atualizado e completo
- Sem histórico anterior de apontamentos

MÉDIO:
- Algumas transações atípicas, mas justificáveis
- Pequenas incompatibilidades com perfil
- Histórico de 1-2 apontamentos anteriores
- Cadastro parcialmente desatualizado

ALTO:
- Transações significativamente incompatíveis
- Padrão de fracionamento identificado
- Múltiplos apontamentos no histórico
- Recusa de atualização cadastral
- Envolvimento de PEP

CRÍTICO:
- Evidências claras de estruturação/fracionamento
- Múltiplas red flags simultâneas
- PEP com transações suspeitas
- Operações em paraísos fiscais sem justificativa
- Cadastro fraudulento ou inconsistente

DECISÕES E STATUS:

Após análise, o analista deve atribuir STATUS ao alerta:

0 - Aguardando Análise: Status inicial do sistema

Status Intermediários (dependem do workflow configurado):
- Análise em andamento
- Aguardando informações da agência
- Encaminhado para Comitê
- Em diligência

Status Finalizados:
- 90 - Informado ao COAF: Comunicado criado automaticamente
- Justificado/Não Procedente: Alerta sem fundamento
- Atípico: Transação atípica mas não suspeita
- Outros status conforme parametrização

REGISTRO DO PARECER:

O parecer deve conter:

1. RESUMO EXECUTIVO (2-3 linhas)
   - Cliente, valor envolvido, período analisado

2. ANÁLISE DETALHADA
   - Regras que apontaram o cliente
   - Transações específicas analisadas
   - Comparação com perfil e médias
   - Identificação de padrões suspeitos

3. INFORMAÇÕES COMPLEMENTARES
   - Dados cadastrais relevantes
   - Histórico de apontamentos
   - Situações especiais (PEP, Funcionário, etc.)

4. CONCLUSÃO E RECOMENDAÇÃO
   - Classificação de risco
   - Status atribuído
   - Ação recomendada (monitorar, comunicar COAF, arquivar)
   - Justificativa da decisão

IMPORTANTE:
- Prazo regulatório para análise: parametrizável no sistema
- Status 90 cria automaticamente Comunicado Atípico ao COAF
- Alerta com status 0 por mais de X dias: movido automaticamente para status 30
- Todas as análises e interações devem ser registradas no Parecer
- Possibilidade de Upload de arquivos (doc, docx, pdf, xls, xlsx)
- Montar Colagem: ferramenta para registrar dúvidas e solicitar informações

PARTE 3: COMUNICAÇÃO AO COAF
===============================================================================

MÓDULO COAF NO SISTEMA DTEC-LD:

TIPOS DE COMUNICADOS:

1. COMUNICADO ATÍPICO
   Usado para operações suspeitas identificadas pelo sistema ou manualmente.

   Criação Automática:
   - Ao atribuir Status 90 no parecer de um alerta
   - Sistema cria comunicado automaticamente
   - IMPORTANTE: Cliente deve ter CPF ou CNPJ válido

   Criação Manual:
   - Menu → COAF → Comunicado Atípico
   - Preencher todos os campos obrigatórios
   - Selecionar enquadramento COAF
   - Definir período de transações

   Campos Obrigatórios:
   - Documento de identificação (CPF/CNPJ)
   - Nome do cliente
   - Período de transações (Data Inicial/Final)
   - Enquadramento COAF
   - Descrição da ocorrência
   - Transações que fundamentam a comunicação

2. COMUNICADO EM ESPÉCIE
   Obrigatório para operações em espécie ≥ R$ 10.000,00

   Como criar:
   - Menu → COAF → Comunicado em Espécie
   - Informar dados da operação
   - Valor, data, cliente envolvido
   - Enviar em até 24 horas da operação

3. COMUNICADO AUTOMÁTICO
   Gerado automaticamente pelo sistema conforme regras pré-estabelecidas.

   Monitoramento:
   - Menu → COAF → Comunicado Automático
   - Verificar comunicados gerados
   - Revisar antes do envio
   - Corrigir erros se houver

FUNCIONALIDADES DO MÓDULO COAF:

1. LISTA OS COMUNICADOS
Consultar comunicados criados:
- Filtros: Data, Status, Cliente, Tipo de Comunicado
- Status possíveis: Rascunho, Enviado, Confirmado, Erro
- Visualização de detalhes completos
- Download de arquivo para envio

2. MANUTENÇÃO DO COMUNICADO
Editar comunicados (antes do envio):
- Alterar enquadramento
- Ajustar período de transações
- Incluir/excluir transações específicas
- Adicionar observações complementares
- Corrigir dados cadastrais

Ações disponíveis:
- Editar: modificar dados do comunicado
- Excluir: remover comunicado (apenas rascunhos)
- Enviar: submeter ao COAF
- Gerar Arquivo: criar arquivo para envio manual

3. PESQUISA DE OCORRÊNCIA
Buscar ocorrências comunicadas:
- Por CPF/CNPJ do cliente
- Por período de comunicação
- Por enquadramento
- Por status do comunicado
- Histórico completo de comunicações

4. IMPORTAR RECIBO
Importar recibos de confirmação do COAF:
- Upload do arquivo de recibo
- Validação automática
- Vinculação ao comunicado original
- Atualização de status para "Confirmado"

5. ERROS DE IMPORTAÇÃO AUTOMÁTICA
Log de erros na criação de comunicados automáticos:
- Identificar comunicados que falharam
- Motivo do erro (campo obrigatório, dado inválido, etc.)
- Permitir correção manual
- Reprocessar após correção

6. PARAMETRIZAÇÃO COAF

A) Enquadramento COAF:
- Cadastrar tipos de enquadramento
- Códigos e descrições conforme COAF
- Exemplos:
  * Operações incompatíveis com patrimônio
  * Operações fracionadas
  * Resistência em fornecer informações
  * Movimentação em paraísos fiscais

B) Erro COAF:
- Catálogo de mensagens de erro
- Orientações para correção
- Vinculação com campos do comunicado

C) Tipo Envolvimento:
- Autor
- Beneficiário
- Intermediário
- Representante Legal

D) Tipo Servidor Público:
- Categorias conforme COAF
- Níveis hierárquicos
- Esferas (Federal, Estadual, Municipal)

7. LOG AUDITORIA (COAF)
Rastreabilidade completa:
- Quem criou o comunicado
- Data e hora de criação
- Modificações realizadas
- Quem alterou
- Quem enviou
- Status de confirmação

ENQUADRAMENTOS COAF (Principais):

1. Operações incompatíveis com patrimônio ou renda
   - Cliente movimenta valores muito superiores ao declarado
   - Faturamento não compatível com porte da empresa

2. Operações fracionadas para evitar registro
   - Smurfing: múltiplas operações abaixo do limite
   - Estruturação: divisão proposital de valores

3. Operações em espécie de valor significativo
   - Depósitos/saques ≥ R$ 10.000 em espécie
   - Padrão frequente de operações em espécie

4. Movimentação de recursos de origem não comprovada
   - Sem fonte identificável
   - Incompatível com atividade econômica

5. Operações com países ou territórios com deficiências
   - Paraísos fiscais
   - Países sob sanção internacional

6. Resistência em fornecer informações cadastrais
   - Recusa de atualização
   - Informações incompletas ou falsas
   - Dificulta verificação de dados

7. Operações realizadas por interposta pessoa
   - Uso excessivo de procurações
   - Laranjas identificados
   - Beneficiário final oculto

PRAZOS REGULATÓRIOS:

Comunicação Obrigatória:
- Operações em espécie ≥ R$ 10.000: até 24 horas
- Operações suspeitas: conforme regulamentação COAF
- Limite pode variar conforme circular vigente

Análise de Alerta:
- Prazo parametrizável no sistema
- Geralmente: 10-30 dias úteis
- Contar da data do apontamento

Envio ao COAF:
- Após conclusão da análise com Status 90
- Verificar dados antes do envio
- Aguardar recibo de confirmação

PROCESSO COMPLETO DE COMUNICAÇÃO:

PASSO 1: Análise do Alerta
- Analista identifica operação suspeita
- Registra parecer detalhado
- Atribui Status 90 - Informado ao COAF

PASSO 2: Criação Automática do Comunicado
- Sistema cria Comunicado Atípico automaticamente
- Replica período de transações do parecer
- Preenche dados básicos do cliente

PASSO 3: Revisão do Comunicado
- Menu → COAF → Lista os Comunicados
- Localizar comunicado criado
- Clicar em "Editar"
- Revisar todos os campos:
  * Enquadramento correto
  * Período de transações adequado
  * Descrição completa da ocorrência
  * Transações listadas corretamente

PASSO 4: Complementar Informações
- Adicionar transações relevantes
- Incluir observações importantes
- Verificar dados cadastrais
- Conferir enquadramento

PASSO 5: Validação Final
- Verificar campos obrigatórios preenchidos
- Conferir valores e datas
- Revisar descrição da ocorrência

CONCEITOS FUNDAMENTAIS DE REGRAS:

O QUE É UMA REGRA?
Conjunto de critérios e condições que, quando atendidos, identificam um cliente como potencial suspeito de lavagem de dinheiro, gerando um alerta para análise.

COMPONENTES DE UMA REGRA:

1. CRITÉRIOS
   - Condições a serem verificadas
   - Exemplo: "Valor total de transações em espécie > R$ 50.000 no mês"

2. OPERADORES LÓGICOS
   - E (AND): todas as condições devem ser atendidas
   - OU (OR): pelo menos uma condição deve ser atendida
   - NÃO (NOT): negação de condição

3. PERÍODO DE ANÁLISE
   - Diário, Semanal, Mensal, Trimestral, Anual
   - Janela móvel ou fixa

4. GRAVIDADE
   - Leve: 2 pontos
   - Média: 3 pontos
   - Grave: 4 pontos

5. PONTO DE CORTE
   - Limiar que define se a regra aponta o cliente
   - Exemplo: valor > X reais, quantidade > Y transações

6. EXCLUSÕES/FILTROS
   - Clientes ou transações que não devem ser considerados
   - Exemplo: excluir funcionários, excluir tipos de transação específicos

TIPOS DE REGRAS:

1. REGRAS ESTÁTICAS
Desenvolvidas pela Softon, baseadas em melhores práticas e tipologias conhecidas.

Características:
- Pré-configuradas
- Atualizadas em releases do sistema
- Testadas e validadas
- Baseadas em regulamentação

Exemplos:
- Operações em espécie acima de limite
- Movimentação incompatível com renda/faturamento
- Transações para paraísos fiscais
- Fracionamento de operações
- Operações por procuração

Gestão:
- Ativar/Inativar: Menu → Parâmetros → Regra
- Alterar gravidade: Leve, Média, Grave
- Ajustar pontos de corte (quando parametrizável)

2. REGRAS DINÂMICAS
Criadas pelo próprio cliente através da interface do sistema, sem necessidade de desenvolvimento.

Características:
- Interface amigável
- Criação sem área de TI
- Testáveis via Simulação
- Importação para produção após validação

Vantagens:
- Atende necessidades específicas da instituição
- Rápida implementação
- Ajustável conforme necessidade
- Não depende de fornecedor

COMO CRIAR UMA REGRA DINÂMICA:

1. Planejar a Regra
   - Definir objetivo: o que se quer identificar?
   - Listar critérios necessários
   - Definir período de análise
   - Estabelecer ponto de corte
   - Determinar gravidade

2. Acessar o Sistema
   - Menu → Regra Dinâmica → Nova Regra

3. Configurar Dados Básicos
   - Nome da regra (descritivo)
   - Descrição detalhada
   - Gravidade (Leve, Média, Grave)
   - Status: Ativo/Inativo

4. Definir Critérios
   - Campo a ser analisado (ex: Valor da Transação)
   - Operador (>, <, =, >=, <=, <>)
   - Valor de comparação
   - Adicionar múltiplos critérios (AND/OR)

5. Configurar Período
   - Período de análise (dias, meses)
   - Janela móvel ou fixa

6. Adicionar Exclusões
   - Filtros de cliente (PEP, Funcionário, Segmento)
   - Filtros de transação (Tipo, Via, Forma)
   - Filtros de agência/região

7. Salvar e Testar
   - Salvar a regra
   - Testar via Simulação antes de produção

EXEMPLOS DE REGRAS DINÂMICAS:

EXEMPLO 1: Transações em Espécie Elevadas
Objetivo: Identificar clientes com múltiplas operações em espécie

Critérios:
- Tipo de Transação = Espécie
- Quantidade de Transações > 10 no mês
- Valor Total > R$ 100.000 no mês
- Operador: E (AND)

Exclusões:
- Ramo de Atividade = Comércio Varejista (volumes esperados)

Gravidade: Média
Período: Mensal

EXEMPLO 2: Movimentação Incompatível com Renda
Objetivo: Clientes Pessoa Física com movimentação acima da renda declarada

Critérios:
- Tipo de Cliente = Pessoa Física
- Valor Total de Créditos > (Renda Declarada × 3) no mês
- Operador: E (AND)

Exclusões:
- PEP = Sim (já monitorado especificamente)
- Funcionário = Sim

Gravidade: Grave
Período: Mensal

EXEMPLO 3: Transações Fracionadas
Objetivo: Identificar possível fracionamento para evitar comunicação obrigatória

Critérios:
- Tipo de Transação = Espécie
- Quantidade de Transações > 5 por dia
- Valor Individual > R$ 8.000 E < R$ 9.900
- Operador: E (AND)

Gravidade: Grave
Período: Diário

EXEMPLO 4: Procurações Atípicas
Objetivo: Clientes com movimentação majoritariamente por procuração

Critérios:
- Percentual de Transações por Procuração > 80%
- Quantidade de Procuradores > 3
- Valor Total Movimentado > R$ 200.000 no mês
- Operador: E (AND)

Gravidade: Média
Período: Mensal

EXEMPLO 5: Operações Internacionais de Risco
Objetivo: Transações para países de risco

Critérios:
- Tipo de Transação = Internacional
- País Destino IN (Lista de Paraísos Fiscais)
- Valor Total > R$ 50.000 no trimestre
- Operador: E (AND)

Exclusões:
- Ramo de Atividade = Importação/Exportação

Gravidade: Grave
Período: Trimestral

FUNCIONALIDADE SIMULAÇÃO:

OBJETIVO:
Testar regras (existentes ou novas) antes de colocá-las em produção, permitindo ajustes sem impactar a base produtiva.

COMO USAR:

1. Acessar Simulação
   - Menu → Simulação

2. Selecionar Lote
   - Escolher lote já processado (para comparação)
   - Ou usar último lote disponível

3. Selecionar Regras
   - Todas as regras (processamento completo)
   - Regras específicas (testar uma nova regra)

4. Ajustar Parâmetros (Opcional)
   - Alterar pontos de corte
   - Modificar conceitos
   - Testar diferentes configurações

5. Executar Simulação
   - Clicar em "Processar"
   - Aguardar conclusão

6. Analisar Resultados
   - Quantidade de alertas gerados
   - Comparar com produção (se aplicável)
   - Verificar clientes apontados
   - Avaliar assertividade esperada

7. Decidir
   - Se satisfatório: importar para produção
   - Se não: ajustar parâmetros e simular novamente

CASOS DE USO DA SIMULAÇÃO:

CASO 1: Testar Nova Regra Dinâmica
1. Criar regra dinâmica
2. Executar simulação com essa regra
3. Analisar clientes apontados
4. Verificar se são realmente suspeitos
5. Ajustar critérios se necessário
6. Simular novamente até validar
7. Importar para produção

CASO 2: Ajustar Ponto de Corte de Regra Existente
Situação: Regra gerando muitos falsos positivos

Ação:
1. Identificar regra problemática (Relatório 01)
2. Simular com ponto de corte mais alto
3. Verificar redução de alertas
4. Avaliar se não está deixando passar suspeitos
5. Aplicar ajuste em produção

CASO 3: Avaliar Impacto de Nova Regulamentação
Situação: COAF alterou limite de comunicação obrigatória

Ação:
1. Ajustar parâmetros das regras afetadas
2. Simular com novo limite
3. Estimar impacto (mais/menos alertas)
4. Planejar capacidade de análise
5. Aplicar mudança

CASO 4: Testar Exclusões/Filtros
Situação: Regra está apontando funcionários do banco

Ação:
1. Adicionar filtro: Funcionário = Sim (excluir)
2. Simular com o filtro
3. Verificar se funcionários foram excluídos
4. Aplicar em produção

ANÁLISE DE EFETIVIDADE DE REGRAS:

Usar Relatório 01 - Índices de desempenho de regra

Indicadores:
- Quantidade de Alertas: volume gerado pela regra
- Comunicações ao COAF: quantos alertas resultaram em comunicação
- Taxa de Assertividade: (Comunicações / Total Alertas) × 100

Interpretação:
- Alta assertividade (>10%): Regra muito efetiva, manter
- Média assertividade (5-10%): Regra adequada, monitorar
- Baixa assertividade (<5%): Avaliar ajustes ou desativar
- Zero comunicações: Considerar desativar (salvo regra nova)

QUANDO AJUSTAR UMA REGRA:

1. Muitos Falsos Positivos
   - Sintoma: Baixa taxa de assertividade
   - Ação: Aumentar ponto de corte ou adicionar filtros

2. Poucos Alertas
   - Sintoma: Regra não aponta há meses
   - Ação: Reduzir ponto de corte ou revisar critérios

3. Mudança Regulatória
   - Sintoma: Nova circular do BACEN/COAF
   - Ação: Ajustar conforme nova regulamentação

4. Mudança no Perfil da Carteira
   - Sintoma: Instituição mudou foco de atuação
   - Ação: Revisar pertinência das regras

5. Feedback dos Analistas
   - Sintoma: Analistas reportam inconsistências
   - Ação: Revisar critérios com equipe

QUANDO DESATIVAR UMA REGRA:

Critérios:
- Taxa de assertividade consistentemente baixa (<2%)
- Regra redundante (outra regra cobre o mesmo cenário)
- Mudança regulatória tornou regra obsoleta
- Custo operacional (análise) não justifica benefício

Processo:
1. Documentar motivo da desativação
2. Validar com equipe e gestão
3. Menu → Parâmetros → Regra → Marcar como Inativa
4. Monitorar por período (pode haver necessidade de reativar)

QUANDO CRIAR UMA NOVA REGRA:

Cenários:
- Nova tipologia de lavagem identificada
- Orientação do COAF/BACEN
- Análise de casos reais revelou gap
- Especificidade da instituição

Processo:
1. Definir tipologia ou padrão a identificar
2. Listar critérios e condições
3. Criar regra dinâmica
4. Simular extensivamente
5. Validar com analistas experientes
6. Importar para produção
7. Monitorar resultados iniciais

AJUSTE DE GRAVIDADE:

Critérios:
- Grave (4 pontos): Forte indicativo de lavagem (ex: fracionamento, paraísos fiscais)
- Média (3 pontos): Situação atípica com risco (ex: movimentação incompatível)
- Leve (2 pontos): Desvio menor do padrão (ex: cadastro desatualizado)

Impacto:
- Afeta pontuação total do cliente
- Influencia priorização de análise

PROCESSAMENTO DE LOTES:

O QUE É UM LOTE?
Conjunto de dados carregados no Repositório Único para processamento.
- Geralmente: dados do dia anterior (D-1)
- Identificação: número sequencial único
- Status: Aguardando, Processando, Concluído, Erro

PROCESSO DE CARGA E PROCESSAMENTO:

1. Carga de Dados
   - Dados extraídos dos sistemas fonte
   - Transformação e validação
   - Carga no Repositório Único
   - Criação do Lote

2. Processamento de Regras
   - Execução de todas as regras ativas
   - Análise de transações do lote
   - Comparação com médias e padrões
   - Identificação de clientes suspeitos

3. Criação de Alertas
   - Clientes apontados por regras
   - Cálculo de pontuação
   - Status inicial: 0 (Aguardando Análise)
   - Distribuição em filas

4. Finalização
   - Lote concluído
   - Disponível para análise
   - Geração de relatórios

TROUBLESHOOTING - PROBLEMAS COMUNS:

PROBLEMA: Lote não processou
Verificar:
1. Menu → Tela Inicial → Painel de Processamento
2. Verificar se lote está em processamento ou aguardando
3. Relatório 30: Acompanhamento de Lote
4. Verificar erros de carga

Possíveis causas:
- Erro na carga de dados
- Regra com erro impedindo processamento
- Falta de recursos (memória, CPU)
- Configuração incorreta

Solução:
- Revisar logs de erro
- Desativar regra problemática se necessário
- Reprocessar lote manualmente

PROBLEMA: Regra foi desativada automaticamente
Indicação:
- Aviso na Tela Inicial
- "Regra X foi desativada por erro"

Causas:
- Regra com lógica inconsistente
- Campo não encontrado
- Divisão por zero
- Configuração inválida

Solução:
1. Verificar aviso na Tela Inicial
2. Corrigir regra: Menu → Parâmetros → Regra
3. Revisar lógica da regra
4. Testar via Simulação
5. Reativar regra

PROBLEMA: Alertas não aparecem na fila
Verificar:
1. Filtros da fila estão corretos?
2. Período de data inclui os alertas?
3. Status correto selecionado?
4. Grupo de Acesso tem permissão?

Solução:
- Ampliar filtros
- Verificar se alertas foram criados (Relatório 02)
- Verificar permissões do usuário

PROBLEMA: Comunicado ao COAF não foi criado automaticamente
Verificar:
1. Menu → COAF → Erros de Importação Automática
2. Cliente tem CPF/CNPJ válido?
3. Período de transações foi informado no parecer?

Solução:
- Corrigir dados do cliente
- Criar comunicado manualmente se necessário
- Verificar log de auditoria

PROBLEMA: Relatório não gera ou demora muito
Causas:
- Período muito extenso
- Muitos registros
- Servidor sobrecarregado

Solução:
- Reduzir período de consulta
- Aplicar filtros mais restritivos
- Aguardar horário de menor uso
- Gerar em lote (agendamento)

PROBLEMA: Performance lenta
Verificar:
1. Quantidade de alertas em aberto
2. Processamento em execução
3. Relatórios sendo gerados
4. Múltiplos usuários simultâneos

Otimizações:
- Limpar alertas históricos (Status 30)
- Processar em horários específicos
- Ajustar configurações de servidor
- Revisar índices de banco de dados
